# 💾 SYSTEM CONTEXT & PRE-PROMPT: CHIPTUNES FANTASY
**Current State: v1.1.1 | Target State: v1.2.0 (The Scene-DJ Refactoring)**

## 1. PROJECT PHILOSOPHY & VISION ("Von Nerds für Nerds")
*Chiptunes Fantasy* is the ultimate 8-bit/16-bit bare-metal music disk emulator running in the browser. 
*   **Zero Dependencies:** 100% Vanilla ES6 JavaScript, HTML5, and CSS3. No React, no Vue, no external libraries.
*   **Pure Mathematics:** No MP3s. We parse original demoscene binary formats (`.SID`, `.YM`, `.MOD`, `.XM`) and render the audio asynchronously via custom low-latency `AudioWorkletProcessors`.
*   **Hardware Accuracy:** We simulate physical chip limits (e.g., Wire-AND pull-downs, JFET triode saturation, 32-step logarithmic DACs, Zero-Order Hold oversampling) in cycle-exact lockstep.
*   **Demoscene Aesthetics:** Visuals strictly adhere to retro hardware limitations (scanline quantization, chunky pixels, strictly 16-color palettes, hardware blending modes). Modern "smooth" WebGL/Canvas blending is forbidden unless it accurately simulates physical CRT artifacts (phosphor decay, beam blooming).

## 2. STRICT PROGRAMMING STYLE GUIDELINES
When writing or refactoring code for this project, the following rules are absolute:
1.  **Performance in Hot Paths:** Inside audio loops or 60FPS render loops, object allocation (`new`, `[]`, `{}`) is strictly forbidden to prevent garbage collection stutter. Use pre-allocated `Float32Array` or `Uint8Array`.
2.  **Bitwise Math over Floats:** Use bitwise operators (`| 0`, `& ~1`, `>>`) for fast integer truncation and masking instead of `Math.floor()` or `Math.round()` where performance is critical.
3.  **Encapsulation:** Visuals and DSP logic must remain modular. Main loops only act as orchestrators. 
4.  **No "Magic Numbers":** Hardcoded values must have comments explaining their physical or historical origin (e.g., `4.42 kHz RC Filter`, `19705 VBLANK cycles`).

## 3. CURRENT ARCHITECTURE (v1.1.1 Arsenal)
### DSP / AudioWorklets
*   **C64 (MOS SID 6581):** 1MHz Lockstep CPU (`cpu6502.js`), 255-Tap Polyphase Sinc-FIR Decimator, True Analog Wire-AND simulation, Sustain-Drop bug emulation.
*   **Amiga 500 (MOS Paula 8364):** 192kHz (4x) Oversampled ZOH DAC to preserve ultrasonic "Amiga Shimmer", L-R-R-L Hard-Panning with 3.5% crosstalk, Word-Aligned DMA loops.
*   **Atari ST (Yamaha YM2149F):** 2MHz True Lockstep, 32-Step Logarithmic DAC, Digidrum DAC Injection (Zero-Click Hijack Isolation), discrete 5-bit hardware envelopes.

### Visuals (Canvas 2D)
All visuals are encapsulated in `/js/visuals/` as ES6 classes:
*   `Oscilloscope`: Physical CRT simulation with offscreen phosphor accumulation, multi-pass tube blooming, and thermal jitter.
*   `FftAnalyzer`: Segmented LED Vacuum Fluorescent Display (VFD) with hardware peak-hold gravity snapping.
*   `Copperbars`: Symmetrical 3D-Helix Z-buffer sorted rasterbars with color banding (12-bit/9-bit/4-bit).
*   `C64Starfield`: Motion-blur warp streaks using the strict 16-color VIC-II palette.
*   `AmigaCube`: 3D Glenz-Vector Cube with backface culling, flat-shading, and audio-reactive vector blooming.
*   `AtariBobs`: Pre-rendered 3D sphere sprites on morphing Lissajous curves.

## 4. MISSION OBJECTIVE: VERSION 1.2.0 (The Scene-DJ)
**Goal:** We are preparing the internal architecture for our **v2.0.0 release**, which will feature dynamic, rule-based visual choreographies. For **v1.2.0**, we will implement the foundational `Demoscene-Sequencer (DSS)` and upgrade our existing visual components into standardized `Demo-Scene-Elements (DSE)`. 

*Constraint for v1.2.0:* No new visual effects will be built. The existing effects (Copperbars, Bobs, Cube, Starfield) will be refactored to fit the new interface. Because there is currently only one effect per computer type, dynamic swapping is not yet active, but the *plumbing* must be perfectly wired.

### Architectural Specs to Implement:

#### A. Demo-Scene-Element (DSE) Interface
Every visual component must be refactored to export standardized metadata and feature a strict internal State Machine.
*   **Metadata Block:**
    *   `name` (String)
    *   `computerType` (Array: e.g., `['c64', 'amiga', 'atari']` or `'all'`)
    *   `placementType` (String: `foreground`, `background`, `floor`, `overlay`)
    *   `energyLevel` (Array: `['idle', 'quiet', 'medium', 'crazy']`)
*   **State Machine (Triggered by DSS):**
    *   `Starting`: Scene blends in (e.g., alpha fade-in, speed ramps up).
    *   `Playing`: Normal showcase of the scene effect.
    *   `Climax`: Scene effect showdown (audio-reactive overdrive).
    *   `Stopping`: Scene gracefully blends out to zero opacity or shrinks away.
    *   `Idle`: Deep sleep mode (skip calculations to save CPU).

#### B. Demoscene-Sequencer (DSS) 'Scene-DJ'
A new orchestrator class (`js/visuals/scene-dj.js`) that takes control away from `visualizer.js`.
*   Monitors the energy of the active music track (via `channelVolumes` and transient analysis).
*   **DSS Rule Set:**
    *   `Min Play Time`: A DSE must run for at least *X* seconds before it can transition out, preventing chaotic flickering.
    *   `Crazy Trigger Level`: If the track's master energy spikes above a threshold, the DSS pushes active DSEs into the `Climax` state.
    *   `Max Parallel DSEs`: Ensures the screen doesn't get cluttered (e.g., max 1 background + 1 foreground element).

#### C. Integration in `visualizer.js`
*   `visualizer.js` loses direct control over the `render()` calls of the gimmicks.
*   It feeds time (`t`) and audio metrics into the `Scene-DJ`.
*   The `Scene-DJ` manages the active DSE pool, updates their state machines, and issues the `render()` commands based on placement Z-order (Background first, Foreground last).
